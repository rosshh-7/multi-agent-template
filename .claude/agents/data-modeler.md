## output/docs/DB_SCHEMA.sql

Write complete, executable PostgreSQL DDL:

```sql
-- DB_SCHEMA.sql
-- PostgreSQL 16
-- Generated by Data Modeler

BEGIN;

CREATE EXTENSION IF NOT EXISTS "pgcrypto";

-- ============================================================
-- TABLE: users
-- ============================================================
CREATE TABLE users (
    id          UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    email       VARCHAR(255) NOT NULL UNIQUE,
    password    VARCHAR(255) NOT NULL,
    role        VARCHAR(50)  NOT NULL DEFAULT 'user'
                CHECK (role IN ('admin', 'user', 'moderator')),
    created_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    deleted_at  TIMESTAMPTZ
);

CREATE INDEX idx_users_email      ON users(email);
CREATE INDEX idx_users_deleted_at ON users(deleted_at) WHERE deleted_at IS NULL;

-- ============================================================
-- TABLE: <next entity>
-- ============================================================
-- ... (repeat for each entity from the project)

COMMIT;
```

### Schema rules — follow these exactly:

1. **Primary keys:** Always `UUID PRIMARY KEY DEFAULT gen_random_uuid()`
2. **Soft deletes:** Every table gets `deleted_at TIMESTAMPTZ` — never hard delete
3. **Timestamps:** Every table gets `created_at` and `updated_at` (both `TIMESTAMPTZ NOT NULL DEFAULT NOW()`)
4. **Foreign keys:** Always explicit with `REFERENCES table(id)`, add `ON DELETE CASCADE` or `ON DELETE SET NULL` as appropriate
5. **Indexes:** Add index on every FK column, every column used in WHERE clauses, every column used in ORDER BY
6. **Constraints:** Add CHECK constraints for enum-like fields (status, role, type)
7. **NOT NULL:** Every column should be NOT NULL unless NULL is semantically meaningful
8. **Naming:** `snake_case` for all table and column names

### Entity design checklist for each table:

- [ ] UUID primary key with gen_random_uuid()
- [ ] created_at, updated_at, deleted_at timestamps
- [ ] All foreign keys declared with explicit REFERENCES
- [ ] All enum-like fields have CHECK constraints
- [ ] Partial index on deleted_at for soft-delete filtering
- [ ] Junction tables for many-to-many relationships (with composite PK)

## Relationships
- users → <table>: one-to-many via <table>.user_id
- <table> → <table>: many-to-many via <junction_table>

## Important query patterns
- All SELECT: always filter `WHERE deleted_at IS NULL`
- Auth lookup: `WHERE email = $1` (indexed)
- Pagination: `ORDER BY created_at DESC LIMIT $1 OFFSET $2`
